# Ownership & Sales Remediation — A8 closure (CoStar Contacts retroactive harvest)

Date: 2026-05-29. After pausing the A9b cutover (safe prep complete; cutover deferred to a dedicated session), picked up the standalone TODO items starting with **A8**.

## Outcome: A8 closed as **N/A (not feasible)** — exactly the plan's gated fallback

A8's plan text (§246) is explicit:
> For sales captured by sidebar in the past 6 months, re-run the contacts extractor over the cached page payloads **(if cache exists; if not, skip — going-forward C2 covers new captures).**

The prerequisite — a cache of raw sidebar/CoStar page payloads — **does not exist.**

### Evidence

- **No page-payload cache.** The only staging tables on dia are Salesforce-sync related (`sf_comp_staging`, `sf_deal_staging`, `sf_listing_staging`, `sf_property_staging`) — none retain raw CoStar page captures. The sidebar pipeline (`sidebar-pipeline.js`) extracts fields and writes **curated** rows; it persists `raw_payload` only into specific curated tables (`deed_records` = the sale object, `parcel_records`/`tax_records` = land/imp values). There is no general inbound-payload store.
- **The CoStar Contacts-tab PII (phone/email/website per buyer/seller/broker) was never persisted pre-C2.** It arrives as `metadata.contacts` — a structure separate from the `sales_history` sale object that `deed_records.raw_payload` retains. So it cannot be recovered from the curated tables, and there is no raw page to re-run the extractor over.
- **Going-forward coverage is live (C2 Part B).** `dia.contacts` carries the C2 columns (`data_source`, `sale_id`, `sale_role`, `website`, `address`); 4 sale-linked contacts with PII across 2 roles already persisted since C2 shipped, growing with each new sidebar sale capture. So new captures are covered — only the retroactive backfill is impossible.

### Why this is the correct close (not a gap)

Retroactively harvesting the past contacts would require either (a) the cached payloads (absent) or (b) re-scraping CoStar for every sale captured in the last 6 months (out of scope, rate-limited, and the data may have changed). The plan anticipated exactly this and specified "skip" when no cache exists. C2 ensures the data is captured prospectively, so the contacts table fills in naturally as sales are re-captured or newly captured.
