## The ask
Build the resolver that works the queued tasks and writes the developer:
1. **Walk the chain to the origin.** For each `trace_ownership_to_developer` task, use
   `v_lcc_ownership_chain_completeness` / `ownership_history` to find the **earliest known owner** of
   the property. For **build-to-suit** properties (`is_build_to_suit=true`) the original/first owner
   is the developer by construction — the strongest, safest signal; resolve those first.
2. **Classify the origin as developer vs not.** Only set `developer` when the earliest owner is a
   plausible development entity (reuse the existing entity guards — junk / operator-family /
   federal-anti-pattern / implausible-person filters; a REIT/operator/agency at origin is NOT a
   developer). Where the chain is incomplete (`missing_segments`), leave it for UW#6's deed
   deep-parse to extend the chain first — don't guess across a gap.
3. **Write + reconcile.** Set `properties.developer` (+ `original_developer` / link the developer
   entity via the BD spine) **fill-blanks only**, through the gov provenance gate
   (`source='chain_resolution'` at a sensible rank), reversible, idempotent. Resolve the task
   (`research_tasks` → resolved) only when a developer is actually written; leave genuinely
   unresolvable chains open (honest).
4. **Feed the BD spine.** A resolved developer is a high-value BD entity (repeat developer across
   multiple properties = a relationship). Ensure the developer entity links into the entity graph /
   portfolio so it surfaces in the priority queue — the chain-to-developer doctrine's payoff.

## Why it matters
Developers are the apex of the gov BD graph — they build-to-suit for agencies, hold portfolios, and
are the highest-value repeat relationships. Identifying them from the ownership chain turns 1,328
dead-queued tasks into named BD targets and feeds the OM/BOV narrative (who developed it). Depends
partly on UW#6 (deed deep-parse) to extend incomplete chains — sequence UW#6 first where chains are
short.
