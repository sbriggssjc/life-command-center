### 1. Gov Core_Cap_Dot — firm-remaining-at-sale (SQL migration)

User's R66 question about over-counting applies to gov too, but via a
different mechanism than dia.

**Diagnosis**:
- `gov.leases.firm_term_years` is a real stored column (16,009 of 16,460 rows).
- It represents the firm (non-cancellable) PORTION of the lease at signing,
  NOT the total lease length and NOT firm-remaining-at-sale. Confirmed
  76% of (firm_term_years, commencement_date, expiration_date) triples
  diverge by >1 year, e.g. a 15-yr total lease commonly carries
  firm_term_years=10 (10-yr firm + 5-yr soft window).
- `cm_gov_core_cap_rate_dots` was returning `l.firm_term_years` verbatim,
  so a sale 4 years into a 10-yr firm lease was over-credited as having
  10 years firm when in reality 6 years remained.

**Audit numbers** (before the fix):
- `cm_gov_core_cap_dot_q`: 886 dots in the 6+ firm-year filter (4-12% cap).
- 410 of 1,101 sales scored 6+ would drop below 6 if we switch to firm-
  remaining-at-sale.

**Fix**: migration `cm_round68_gov_core_cap_dot_firm_remaining_at_sale`
on gov project (`scknotsqkcheojiaewwh`). Replace `l.firm_term_years` with
`l.firm_term_years − GREATEST(0, years_elapsed_from_commencement_to_sale)`
inside the subquery. Per-sale value now reflects what was firm-remaining
at sale.

**Impact**: gov `cm_gov_core_cap_dot_q` drops from 886 → 585 dots
(34% reduction). The 301 dropped sales had been credited with firm-at-
signing but had already burned through enough firm time to fall below
the 6-year threshold by sale date.

**Backout**: replace the GREATEST(...) subtraction with bare
`l.firm_term_years` to restore the old behavior.

**Dia twin**: R67 already fixed dia via a different mechanism (15-yr
total-term cap on the picked lease, because dia.leases doesn't store
firm_term_years — extensions overwrite `lease_expiration` in place).

### 2. Bid_Ask spread chart — wire stacked-line + upDownBars visual

User wanted the master's "dots/lines stacked above the last asking cap
rate over the TTM with drop-down lines between data points" idiom.

**Diagnosis**: R50 (2026-05-22) already added the template — a stacked
2-line chart (Last Ask Cap + Spread) plus chart-level `<c:upDownBars/>`.
But it gracefully fell back to a single-line chart of the spread alone
when the `avg_last_ask_cap` column was missing. And it WAS missing:
`cm-excel-export.js` `bid_ask_spread` CHART_COLUMNS only listed
period_end, subspecialty, avg_bid_ask_spread, pct_price_change — no
avg_last_ask_cap, even though both `cm_dialysis_bid_ask_spread_m` and
`cm_gov_bid_ask_spread_m` return it.

**Fix**: added `avg_last_ask_cap` to the bid_ask_spread CHART_COLUMNS
array (`api/_shared/cm-excel-export.js:530`). On next export, the data
tab now carries the column AND the R50 chart template finds it via
`findCol('avg_last_ask_cap')` and emits the stacked-line + upDownBars
visual matching the master.

**Verified emission shape**:
```
type: multi-line
lineGrouping: stacked
upDownBars: true
series: [
  { valCol: D (avg_last_ask_cap), color: 62B5E5 (sky) },  // baseline
  { valCol: C (avg_bid_ask_spread), color: 003DA5 (navy) }, // stacked on top
]
```

### 3. Legacy Charts tab — honest label

User: "the charts in the individual tabs don't match what's in the
aggregate Charts tab. Let's make sure we are consistent so we can always
copy/paste or extract the same layout and formatted and data from these
charts no matter where we look."

**Diagnosis**: three separate rendering paths:
1. Native Excel chart injector → `Data_*` tabs (R34-R68 fidelity).
2. QuickChart PNG renderer → aggregate `Charts` tab — original, hasn't
   kept up with R34-R68 improvements.
3. LCC app capital markets tab — JavaScript client-side, separate.

**Fix**: updated the Charts tab's subtitle to honestly label its
provenance and direct users to the authoritative Data_* tabs. Full
unification (regenerating Charts tab via native injector, or dropping
the tab entirely) deferred — needs user sign-off on either approach.

Old label:
> Auto-rendered via QuickChart from 34 chart configs. For full chart
> parity, see the MasterPasteReady tab + your master XLSX.

New label:
> LEGACY SNAPSHOT — auto-rendered via QuickChart from N chart configs
> (may be out of sync with the per-tab native charts). For authoritative
> formatting and current data, see the Data_* tabs. For paste-into-master
> workflows, see the MasterPasteReady tab.
