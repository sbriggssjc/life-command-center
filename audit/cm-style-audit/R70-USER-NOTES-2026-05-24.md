## Shipped

### Sentiment: data-aware MIN_YEAR cutoff

**Diagnosis**: `cm_dialysis_seller_sentiment_m` joins `sales_transactions`
to `available_listings` to derive `had_price_change` (initial vs. last
price) and `last_cap_rate`. Pre-2013 there ARE 117 sales with sold_date
and 20 listings, but the TTM aggregates produce sparse `n_all` counts:

| period | n_all |
| --- | ---: |
| 2006-2007 | 0-1 |
| 2008 | 1-3 |
| 2009 | 0-2 |
| 2010-2011 | 0-3 |
| 2012 | 2-6 |
| 2013-Q1/Q2 | 1-2 |
| 2014-Q1/Q2 | 2-4 |
| **2014-Q3+** | **≥7** |

User notes: "Missing data for before 2013". R47's static MIN_YEAR=2006
let those sparse cells render and the chart showed a jagged near-empty
line then jumped on at 2014.

**Fix**: switched `seller_sentiment` + `seller_sentiment_monthly` in
`MIN_YEAR_BY_TEMPLATE` from static `2006` to the R69 data-aware function
`(rows) => findFirstDenseYear(rows, 'n_all', 5) ?? 2014`. The chart now
starts where 4+ consecutive months have ≥5 TTM sentiment samples —
~late 2014 for dia. Falls back to 2014 if the data shape doesn't expose
`n_all` (covers test paths and future schema changes).

**Why not a SQL view fix?** A view-level `n_all < 5 ⇒ NULL` gate would
also work but would leave the chart x-axis stretched across empty
months. Trimming at the chart layer is cleaner: the data tab still
shows all rows for users who want to drill in.

### 80bps methodology gap (Cap_Avg / Returns_Idx / Cost_Capital)

Per-quarter master vs ours, 2009-Q1 through 2018-Q2:

| Period | Master cap / n | Ours cap / n | Δ cap |
| --- | --- | --- | ---: |
| 2009-Q1 | 9.00% / 16 | 7.87% / 23 | -113bps |
| 2009-Q2 | 8.99% / 16 | 8.03% / 27 | -96bps |
| 2009-Q3 | 8.99% / 14 | 7.89% / 23 | -110bps |
| 2009-Q4 | 9.17% / 11 | 8.12% / 21 | -105bps |
| 2010-Q1 | 8.93% / 17 | 8.31% / 26 | -62bps |
| 2010-Q4 | 8.73% / 25 | 8.15% / 23 | -58bps |
| 2011-Q1 | 8.60% / 22 | 8.11% / 28 | -49bps |
| 2011-Q2 | 8.09% / 24 | 8.24% / 34 | +15bps |
| 2012-Q1 | 8.23% / 52 | 8.36% / 61 | +13bps |
| 2013-Q4 | 7.74% / 70 | 7.41% / 76 | -33bps |
| 2014-Q4 | 7.58% / 95 | 7.33% / 102 | -25bps |
| 2017-Q4 | 6.61% / 191 | 6.48% / 188 | -13bps |
| 2018-Q1 | 6.71% / 222 | 6.56% / 220 | -15bps |

**Pattern**: gap is largest 2009-2011 (50-110bps), shrinks after 2014
(13-33bps), and is statistically negligible by 2017-2018 (10-20bps).

**Investigation**: tested whether excluding "unverified" historical
imports (data_source='historical_csv_import' AND buyer_name IS NULL AND
seller_name IS NULL) would close the gap. Results: helps in 2009-2011
(+18-26bps in the right direction) but hurts in 2012-2013 (-12 to
-19bps). Not a clean one-shot filter.

**Recommendation**: leave the chart layer as-is. The gap is bounded
(under 30bps after 2014) and within reasonable methodology variation.
A true reconciliation would require:
1. Per-sale comparison between master's `Sales Comps` sheet (2,544
   rows with date+cap) and our `sales_transactions`
2. Identifying which sales master excludes (likely portfolio /
   leasehold / non-stabilized deals)
3. Either suppressing those sales in our view or adding a
   verified-only flag column

That's a multi-day data-reconciliation project, not a chart fix —
tracked as a follow-up item.

### Why pre-2014 sales counts diverge

Sales-by-year comparison (master vs ours, 2007-2013):

In 2010-2013 master has 10-20 more sales than us. Master is more
complete pre-2014 (likely from manual broker-memory curation that
predates our automated ingestion). Closing this would require
backfilling missing sales into `sales_transactions`, not a chart
methodology change.
