# R71 — Aggregate "Charts" tab regenerated via native injector

User notes 2026-05-23 batch 6: _"the charts in the individual tabs
don't match what's in the aggregate Charts tab. Let's make sure we are
consistent so we can always copy/paste or extract the same layout and
formatted and data from these charts no matter where we look."_

R68 acknowledged the divergence and updated the Charts-tab subtitle to
honestly label it as a "LEGACY SNAPSHOT". R71 fixes the underlying
problem: the aggregate Charts tab now hosts NATIVE Excel charts
live-linked to the Data_* tabs.

## What changed

### Infrastructure: multi-chart-per-sheet support in `injectNativeCharts`

Pre-R71 the injector emitted one drawing.xml per chart and silently
dropped subsequent injections targeting the same sheet (Excel requires
at most one `<drawing r:id="..."/>` per worksheet, so a second injection
would either no-op or conflict).

R71 refactor:
- Inputs collated by `tabName` (the OUTER injection field — where to
  anchor the drawing).
- One `drawing.xml` per sheet, containing N `<xdr:twoCellAnchor>`
  blocks (one per chart on that sheet).
- One `drawing.xml.rels` file per sheet with N chart relationships.
- One sheet rels addition (`<Relationship ... drawing.xml/>`) per
  sheet, regardless of how many charts.
- `<xdr:cNvPr id>` values sequenced (2, 3, 4, ...) so they're unique
  within the parent drawing — Excel requires this.

Single-chart sheets (the Data_* tabs) use the legacy `buildDrawingXml`
helper unchanged so the byte-for-byte XML shape is preserved — existing
snapshot-style tests still pass.

### Aggregate Charts tab: native twins of the Data_* charts

In `cm-excel-export.js`, after `nativeInjections` is fully built for
the Data_* tabs, R71 emits a TWIN injection per template for the
Charts tab:

```js
nativeInjections.push({
  tabName: 'Charts',                 // anchor on aggregate Charts sheet
  spec: { ...originalSpec, anchor }, // data refs unchanged (point to Data_*)
});
```

The key insight: `spec.tabName` (used by chart builders to construct
`'Data_X'!$A$5` data refs) is preserved, while the outer
`injection.tabName` (used by the injector dispatcher to attach the
drawing) is 'Charts'. So the twin chart on the Charts tab reads its
data live from the same Data_* tab as the per-tab chart — edits in
the data tab update both views.

Tiled anchor calculation (matches the legacy PNG layout 1:1 so the
visible page geometry is unchanged):
```
col0=1   (B, 0-indexed)
row0=cursor
col1=14  (O exclusive, ~900px wide matching the legacy PNG)
row1=cursor + 25
cursor += 27 (header + 25 rows + 1 spacer)
```

### Path selection: native-only OR all-PNG, never hybrid

Excel allows at most one drawing per sheet. `ExcelJS.addImage()` creates
its own `<drawing/>` element when called, which would collide with
R71's native chart drawing. So the Charts tab takes one path or the
other:

| Eligible native count | Path |
| --- | --- |
| ≥ 1 (typical post-R34 export) | Pure native. Templates without a native twin are skipped (header only, no chart) — their data tabs are authoritative. |
| 0 (legacy mode, no migrated templates) | Pure PNG. Same as the pre-R71 fallback. |

The subtitle text adapts to which path was taken so users know what
they're looking at.

### Tests

- `R71: injectNativeCharts emits ONE drawing.xml per sheet when multiple
  charts target it` — two injections targeting 'Data_X', verifies one
  drawing file, two chart files, two anchors, unique cNvPr ids, one
  `<drawing/>` element on the sheet.
- `R71: single-chart path preserves legacy drawing.xml byte shape` —
  pre-existing single-chart byte shape (cNvPr id=2, name="Chart 1") is
  unchanged.

End-to-end smoke verified by building a real workbook with 2 native
templates + chartImages, injecting, then inspecting the result:
- Charts sheet has exactly one `<drawing r:id="rId1"/>` element.
- That drawing has 2 anchors with cNvPr ids 2, 3.
- Two new chart XMLs (chart3.xml, chart4.xml) reference
  `'Data_Volume_TTM'!$C$5:$C$136` and `'Data_Cap_Avg'!$C$5:$C$136`
  respectively — same data ranges as their Data_*-anchored siblings.

172/172 tests pass (167 native-chart-injector + 5 export-bundle-audit).

## What the user will see in a fresh post-R71 export

1. Open the workbook.
2. Each `Data_*` tab has its native Excel chart (unchanged from R66+).
3. The aggregate `Charts` tab now has those same native charts tiled
   vertically, B-O columns, 25 rows tall each, with the chart name
   in the header row above. Editing a data cell in any Data_* tab
   updates both that tab's chart AND the matching tile on the Charts
   tab — they're the same live-linked chart object pointed at the
   same data ranges, just anchored on different sheets.
4. Right-clicking any chart on the Charts tab → Select Data → reveals
   the live cell references back to the Data_* tab.
5. The subtitle on the Charts tab reads "N native Excel charts — fully
   editable, live-linked to the Data_* tabs" (where N = count of
   migrated templates, currently 30+ of 34).

## Backout

If the native Charts tab needs to be reverted (e.g. a downstream
consumer expects PNG-only):

1. In `cm-excel-export.js`, remove the `useNativePath` branch and
   restore the pure-PNG loop.
2. The `injectNativeCharts` multi-chart-per-sheet capability is
   harmless on its own — it gracefully handles the single-chart
   case via the byte-preserving legacy path. Safe to leave in place.

## Out of scope

- The QuickChart PNG renderer (`cm-chart-image-renderer.js`) is still
  used to generate the legacy PNGs that drive the chart-name list.
  We could drop the PNG generation entirely now that the native path
  handles the Charts tab, but the PNG rendering also feeds (a) the
  legacy fallback path here and (b) any downstream consumers (PDF
  marketing decks, slide exports) that might still want raster images.
  Deferred — separate decision.

- The Master Excel reference workbook (`Dialysis Comp Work MASTER.xlsx`)
  still has manually-curated Charts whose styling we don't fully match.
  R66 audit found a delta (chart titles, font choices). The R71
  Charts tab uses R34-R68 styling which is the brand-aligned modern
  version. Master parity is a separate ongoing thread.
