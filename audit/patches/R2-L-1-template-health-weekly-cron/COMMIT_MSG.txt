audit(R2-L-1): schedule template health weekly rollup + persist history

Round 2 finding R2-L-1 (HIGH). evaluateTemplateHealth() was wired and
exposed at /api/operations?_route=draft&action=health, but no scheduled job
called it. The weekly health report was regenerated by hand and the loop
had no time-series.

Three coordinated pieces:

1. LCC Opps: new public.template_health_history table. One row per rollup,
   captures the full structured evaluations payload as JSONB plus
   pre-rolled aggregates (template_count, evaluated_count,
   needs_revision_count, stale_count, total_sends, revisions_flagged,
   run_duration_ms). Indexed on recorded_at DESC plus a partial index on
   needs_revision_count > 0.

2. api/operations.js: new POST ?_route=draft&action=health-rollup handler.
   Same evaluateTemplateHealth + flag-and-suggest pass as ?action=health,
   plus persists a row to template_health_history. persist=false query
   param lets UI dashboard tabs call without polluting the time-series.

3. pg_cron lcc-template-health-rollup: Monday 06:00 UTC, POSTs the new
   endpoint with {lookback_days: 120, persist: true}. Goes through the
   standard lcc_cron_post('/api/...', payload, 'vercel') helper.

Sandbox note: api/operations.js (200 KB) was truncated by the first Edit
attempt (under the documented 500KB threshold but still lost the last 11
lines). Restored from git HEAD and re-applied via Python-via-bash with
explicit CRLF preservation. Final git diff --stat: 74 insertions, 0
deletions.

Without D-3's record_send writer, the first N rollups will show
total_sends=0 — that's the correct baseline. The history table starts
capturing the moment template_sends rows land.
