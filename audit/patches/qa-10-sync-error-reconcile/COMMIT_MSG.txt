qa(10): reconcile sync error count across Pipeline/Sync Health/Metrics

Three surfaces disagreed about the same conceptual stat:
  Pipeline page header  "⚠ 1 connector failing: outlook"   → 1
  Sync Health "Errors"  "0 unresolved sync issues"          → 0
  Metrics "Sync Errors" "0 connectors"                      → 0

Root cause: two different concepts were conflated under the same
label. Pipeline counts connector accounts currently in status='error'
or 'degraded' (summary.error from /api/sync?action=health). Sync
Health and Metrics tiles were counting unresolved rows in the
sync_errors log table — a different audit-trail concept that can
diverge from connector status in both directions.

For the operator, the live connector status is the actionable signal
(OAuth expired, transient API failure, etc.). The sync_errors log
table is more of an audit trail. There's already a "Recent Errors"
widget on Sync Health that surfaces the log rows for diagnostics —
having a count tile that disagreed with the Pipeline banner just
made the dashboard look broken.

Fix: two-line change in ops.js.

  1. Sync Health page "Errors" summary tile
       was: unresolvedErrors.length, "unresolved sync issues"
       now: summary.error || 0,      "connectors in error state"

  2. Metrics page "Sync Errors" tile
       was: c.sync_errors || 0,
       now: syncHealthRes.data?.summary?.error || 0
            (renderMetricsPage already fetches /api/sync?action=health
             for the Operational Signals section — just reuses it)
            falls back to c.sync_errors if sync-health failed.

After (verified live):
  Pipeline banner            1 (unchanged)
  Sync Health Errors tile    1 (was 0)
  Metrics Sync Errors tile   1 (was 0)

All three read from summary.error now.
