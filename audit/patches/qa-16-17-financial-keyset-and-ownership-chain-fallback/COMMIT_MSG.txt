qa(16+17): clinic_financial_estimates keyset + gov ownership-chain fallback

Two P0 issues surfaced by QA pass #2 (post-QA-15 fresh walkthrough).
Console showed both errors on every page reload:

  diaQuery clinic_financial_estimates: HTTP 500
    (Postgres 57014 — canceling statement due to statement timeout)
  govQuery v_ownership_chain: HTTP 400
    (column v_ownership_chain.property_id does not exist)

QA-16 — dia clinic_financial_estimates statement_timeout

  36,538 is_latest rows lazy-paginated 1000-at-a-time with OFFSET.
  Page 30 alone took 1,356 ms; later pages tripped statement_timeout.
  count=false was already set — pure OFFSET-seek cost.

  Fix:
    1. New partial keyset index
       idx_cfe_latest_keyset ON clinic_financial_estimates(estimate_id)
       WHERE is_latest = true
    2. dialysis.js lazy loader switched from OFFSET to keyset pagination
       (order=estimate_id.asc, filter2=estimate_id=gt.<last_seen>).
       100-iteration safety cap.

  Verified live: representative page now executes in 4.5 ms
  (was 1,356 ms — ~300× speedup). Full 37-page load ≈ 170 ms total
  (was ~24 s and frequently timing out mid-load).

QA-17 — second v_ownership_chain caller still using property_id on gov

  QA-08 fixed _udOwnerBeginProspecting (detail.js line ~5620) but
  missed a second caller in the main fetch path. detail.js line ~228
  had `leaseNumber ? lease_number=eq.X : mainFilter` — when
  leaseNumber is null and db='gov', the fallback was
  property_id=eq.X, which 400s on gov (column doesn't exist).

  Fix: on gov, no fallback — skip the chain fetch when leaseNumber
  is missing and return { data: [], count: 0 }. No useful chain rows
  exist for a non-leased gov property anyway.
